## 一、项目概览

**AI News Bot** 是一个全自动化的 AI 资讯采集、智能处理、多渠道推送系统。核心能力覆盖：RSS/GitHub 数据采集、LLM 驱动的摘要/评分/分类、向量语义服务、多平台消息推送、定时日报/周报、AI 对话 Agent，并配套 Web 管理面板。

### 技术栈总览

| 维度 | 技术选型 |
|------|---------|
| **后端语言** | Python 3.11+ |
| **Web 框架** | FastAPI 0.135 + Uvicorn |
| **ORM** | SQLAlchemy 2.0 (async) + aiosqlite |
| **数据库** | SQLite（WAL 模式，30s busy_timeout） |
| **数据库迁移** | Alembic |
| **任务调度** | APScheduler 3.11 (AsyncIOScheduler) |
| **向量数据库** | ChromaDB（适配器模式，支持 Milvus/Qdrant 扩展） |
| **Embedding** | Ollama / OpenAI / SiliconFlow / OpenRouter |
| **LLM 集成** | 智谱 / 硅基流动 / OpenRouter / ModelScope / Ollama / OpenAI |
| **AI Agent** | DeepAgents 0.4.11 + LangGraph |
| **前端** | Vue 3.5 + TypeScript + Element Plus 2.13 + Pinia + ECharts |
| **构建工具** | Vite 8 + vue-tsc |
| **部署** | Docker + Docker Compose + Nginx |
| **License** | Mulan PSL v2 |

### 3.2 分层设计

**API 路由层** — 接收 HTTP 请求，参数校验，返回统一响应格式。路由按模块拆分在各 `app/api/` 文件中。

**服务层** — 核心业务逻辑，分为 7 大模块（采集、处理、推送、调度、向量、Agent、RSSHub 集成）。

**数据层** — SQLite 关系数据库（SQLAlchemy ORM）+ ChromaDB 向量数据库，通过适配器模式支持扩展。

### 3.3 关键设计模式

| 模式 | 使用位置 | 说明 |
|------|---------|------|
| **适配器模式** | `vector/adapters/` | 统一向量数据库接口，支持 ChromaDB/Milvus/Qdrant 切换 |
| **单例模式** | `vector_db_manager`, `llm_manager`, `rss_fetcher`, `config_sync_service` | 全局共享实例，避免重复初始化 |
| **工厂模式** | `dynamic_base.create_notifier`, `BackendFactory` | 根据类型动态创建通知器/后端实例 |
| **策略模式** | `processor/` | 不同 LLM Provider 的策略切换 |
| **装饰器模式** | `task_decorator.py` | 定时任务状态追踪和重试逻辑 |
| **注册模式** | `dynamic_base.register_notifier` | 通知器类型注册和自动发现（WeCom/Telegram/Discord/Obsidian） |
| **信号量限流** | `llm_manager`, `batch_processor` | asyncio.Semaphore 控制 LLM 和批量处理并发数 |
| **分层异常** | `middleware/exception_handler.py` | ApiException → NotFound/Unauthorized/Forbidden/Validation |

## 五、核心执行流程

### 5.1 AI 资讯采集 -> 处理 -> 推送 完整流程

```
┌────────────┐   ┌──────────────┐   ┌───────────────┐   ┌────────────┐
│ Scheduler  │   │  RSSFetcher  │   │ ContentFetcher │   │Deduplicator│
│ 定时触发   │──→│  解析RSS源   │──→│  正文补全      │──→│  去重检查   │
│(30分钟一次)│   │              │   │ (trafilatura)  │   │            │
└────────────┘   └──────────────┘   └───────────────┘   └──────┬─────┘
                                                               │
                                                               ▼
┌────────────┐   ┌──────────────┐   ┌───────────────┐   ┌────────────┐
│  Notifier  │←──│  Scheduler   │←──│ BatchProcessor│←──│  入库      │
│ 多渠道推送 │   │ 定时/手动触发 │   │ LLM批量处理   │   │ Status=   │
│ 日报/周报  │   │ 生成报告      │   │ 摘要+评分+标签 │   │ PENDING   │
└────────────┘   └──────────────┘   └───────────────┘   └────────────┘
```

**详细步骤：**

1. **采集阶段**：APScheduler 每 N 分钟触发 `fetch_ai_news` 任务
2. **RSS 解析**：`RSSFetcher.fetch()` 遍历所有启用的 RSS 源，使用 feedparser 解析
3. **内容补全**：对未配置跳过的域名，使用 trafilatura/BeautifulSoup 获取文章正文（8s 超时）
4. **去重检查**：`Deduplicator` 基于标题相似度（difflib）进行去重，可选启用向量语义去重
5. **入库**：文章写入 `articles` 表，状态为 `PENDING`
6. **LLM 处理**：`process_pending_content` 每 5 分钟运行一次，批量处理 PENDING 文章
7. **批量处理**：`BatchProcessor.process()` 每批 5 篇文章，调用 LLM 生成摘要+关键词+标签+评分
8. **智能降级**：批次失败时自动降级为逐篇调用
9. **状态更新**：处理完成后文章状态变为 PROCESSED
10. **报告生成**：`send_daily_report` / `send_weekly_report` 定时任务执行
11. **内容生成**：`ReportContentGenerator` 从数据库查询文章和 GitHub 项目，生成 Markdown 报告
12. **LLM 增强**：可选调用 LLM 翻译英文描述、优化摘要
13. **多渠道推送**：`NotificationManager` 遍历所有启用的 Webhook 配置，按平台格式化后推送

### 5.5 向量服务 7 大场景

| 场景 | 方法 | 说明 |
|------|------|------|
| S1: 语义去重 | `deduplicate()` | 入库前检查相似文章，相似度 > 阈值则跳过 |
| S2: LLM 缓存 | `check_llm_cache()` | 相似查询命中时直接返回缓存结果 |
| S3: 语义搜索 | `search()` | 关键词 + 语义混合搜索 |
| S4: Agent RAG | `retrieve_for_agent()` | Agent 对话时检索相关资讯 |
| S5: 相似推荐 | `recommend()` | 推荐与指定文章相似的内容（阈值 0.80） |
| S6: 主题聚类 | `cluster()` | HDBSCAN 聚类，生成主题标签 |
| S7: GitHub 相似 | `find_similar_repos()` | 查找相似 GitHub 项目 |

所有场景均支持优雅降级 — 向量库不可用时自动退化到基础功能。

### 5.6 评分体系（Scorer）

文章价值评分基于 5 个维度，由 LLM 自动生成 JSON 格式评分结果：

| 维度 | 权重 | 说明 |
|------|------|------|
| 技术创新性 | 25% | 是否有新技术/新方法/新突破 |
| 商业/战略价值 | 25% | 投融资、并购、产品发布、财报影响 |
| 行业影响力 | 20% | 对科技生态的影响范围、社区讨论度 |
| 时效性/稀缺性 | 20% | 独家报道、首发优势、紧急动态 |
| 实用价值 | 10% | 对开发者/从业者的参考价值 |

**评分分级**：≥85 分即时推送 | 60-85 分加入日报 | <60 分低价值

**去重机制**：`Deduplicator` 使用三层去重策略：
1. URL 哈希（BLAKE2b-128）精确匹配
2. 标题指纹（SHA256 标准化后哈希）模糊匹配
3. 向量语义去重（可选的 S1 场景）

数据库写入采用 `IN` 子句分片（500 一批）和指数退避重试（3 次）。

### 6.3 配置加密

敏感配置（API Key、Webhook Secret 等）使用 AES-256-GCM 加密后存入数据库：

- **加密前缀**：`ENC:` + base64(iv + ciphertext + tag)
- **核心函数**：`encrypt_config_value()` / `decrypt_config_value()` / `is_encrypted_value()`
- **向后兼容**：无 `ENC:` 前缀的值按明文处理（兼容旧配置数据）
- **加密密钥**：由应用 `SECRET_KEY` 派生

## 十四、关键风险与注意事项

1. **SQLite 并发限制**：使用 WAL 模式 + busy_timeout=30000 优化，但仍不适用于高并发写入场景
2. **LLM API 限流**：通过信号量（Semaphore）控制并发，每个 Provider 独立配置最大并发数
3. **向量库健康检查**：每 30 秒巡检一次，自动降级和恢复
4. **API Key 安全**：LLM 和 Embedding 的 API Key 加密后存入数据库
5. **企业微信加密**：使用 WXBizMsgCrypt SDK 保障消息安全性
